Images are available on two registries. The steps below walk you through choosing a registry, pulling an image, running it, and pinning to a specific version.

<Note>
  **GitHub Container Registry is recommended.** It has higher rate limits than Docker Hub and is the canonical source for all images in this project.
</Note>

<Steps>

<Step title="Choose a registry">

All images are published simultaneously to both registries under the same tag scheme.

<Tabs>
  <Tab title="GitHub Container Registry (recommended)">
    ```bash
    # Pull from ghcr.io
    docker pull ghcr.io/jauderho/<image>:latest
    ```

    Images at `ghcr.io/jauderho/` are the canonical source. Use this registry to avoid Docker Hub rate limits, especially in CI.
  </Tab>
  <Tab title="Docker Hub">
    ```bash
    # Pull from Docker Hub
    docker pull jauderho/<image>:latest
    ```

    Docker Hub images are published at [hub.docker.com/u/jauderho](https://hub.docker.com/u/jauderho/) and are identical to the GHCR images.
  </Tab>
</Tabs>

</Step>

<Step title="Pull an image">

Pick one of the 80+ available tools. A few common examples:

<CodeGroup>

```bash nginx
docker pull ghcr.io/jauderho/nginx:latest
```

```bash trufflehog
docker pull ghcr.io/jauderho/trufflehog:latest
```

```bash terraform
docker pull ghcr.io/jauderho/terraform:latest
```

```bash ansible
docker pull ghcr.io/jauderho/ansible:latest
```

</CodeGroup>

</Step>

<Step title="Run the image">

Every image follows standard Docker conventions. Most accept the same CLI flags as the upstream tool.

<CodeGroup>

```bash nginx
docker run --rm -p 80:80 ghcr.io/jauderho/nginx:latest
```

```bash trufflehog
docker run --rm ghcr.io/jauderho/trufflehog:latest \
  github --repo https://github.com/example/repo
```

```bash terraform
docker run --rm -v "$(pwd)":/workspace \
  -w /workspace \
  ghcr.io/jauderho/terraform:latest \
  init
```

```bash ansible
docker run --rm -v "$(pwd)":/workspace \
  -w /workspace \
  ghcr.io/jauderho/ansible:latest \
  ansible --version
```

</CodeGroup>

</Step>

<Step title="Pin to a specific version">

Using `:latest` is convenient for development, but production deployments should pin to a specific version or digest.

**Option 1 — Version tag**

Version tags match the upstream tool's release version:

```bash
docker pull ghcr.io/jauderho/terraform:v1.12.1
```

**Option 2 — SHA-256 digest (most reproducible)**

A digest reference is immutable. Even if `:latest` is updated, the digest always resolves to the exact same image layer set:

```bash
# Pull by digest
docker pull ghcr.io/jauderho/terraform@sha256:<digest>

# Use in a Dockerfile
FROM ghcr.io/jauderho/terraform@sha256:<digest>
```

<Tip>
  Get a digest for any tag with `docker inspect --format='{{index .RepoDigests 0}}' ghcr.io/jauderho/terraform:latest`.
</Tip>

This is the same approach used internally — every `FROM` line in these Dockerfiles pins base images by digest to prevent supply chain attacks.

</Step>

<Step title="Verify the image signature">

All images are signed with [cosign](https://github.com/sigstore/cosign). Verifying the signature confirms the image was built by this project and has not been tampered with.

First, install cosign if you do not already have it:

```bash
# Install cosign (see https://docs.sigstore.dev/cosign/system_config/installation for all options)
brew install cosign          # macOS
go install github.com/sigstore/cosign/v2/cmd/cosign@latest  # Go
```

Then verify the image:

```bash
cosign verify ghcr.io/jauderho/nginx:latest \
  --certificate-identity-regexp="https://github.com/jauderho/dockerfiles" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com"
```

A successful verification prints the certificate and signature metadata. A failure means the image did not come from this project's GitHub Actions pipeline.

</Step>

</Steps>

## What makes these images different?

These images are not just re-tagged upstream images. Each one is:

- **Compiled from source** with hardening flags (`-fstack-protector-strong`, `-D_FORTIFY_SOURCE=3`, full RELRO)
- **Built with multi-stage Dockerfiles** so build tools never ship in the final image
- **Scanned daily** by Trivy and Anchore for CVEs
- **Rebuilt automatically** within an hour of a new upstream release tag
- **Signed** with cosign and include an attached SBOM

See the [Security Model](/security) page for the full breakdown.
